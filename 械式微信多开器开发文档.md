## 1. 项目概述

### 1.1 项目背景

微信作为国民级即时通讯工具，在日常办公和生活中不可或缺。然而微信PC客户端默认采用**单实例运行机制**，同一时间仅允许一个登录会话。对于需要同时管理多个微信账号的用户（如电商运营、客服团队、社群管理者等），频繁切换账号严重影响了工作效率。

### 1.2 产品定位

**械式微信多开器** 是一款面向Windows桌面平台的微信多开管理工具，核心功能包括：

- **一键多开**：点击一次即可打开多个微信登录窗口，无需退出已登录的微信
- **独立记忆**：每个窗口拥有独立的数据目录，微信自身的"记住账号"功能使同一实例下次打开时自动展示上次登录的账号头像，用户仅需点击"登录"确认即可
- **动态追加**：即使已打开一个微信，也能通过程序继续打开新的登录窗口
- **实例管理**：可视化查看所有运行中的微信实例，支持添加标签备注区分不同用途
- **轻量高效**：安装包小于10MB，内存占用极低

### 1.3 目标用户

| 用户群体 | 使用场景 |
|---------|---------|
| 电商运营人员 | 管理多个店铺客服账号 |
| 社群运营者 | 维护不同微信群组 |
| 企业经营者 | 区分工作号与生活号 |
| 销售人员 | 多客户线索并行跟进 |
| 普通用户 | 工作与生活账号分离 |

经过调研，现有产品存在以下不足：

1. **商业软件普遍收费**，且闭源无法审计安全性
2. **开源项目功能单一**，缺乏账号记忆、分组管理等实用功能
3. **多数工具不支持"动态追加"**——即已打开微信后继续开新窗口
4. **UI体验较差**，缺乏现代化界面设计
5. **版本兼容性差**，微信更新后频繁失效

**械式微信多开器**将填补这一空白：**免费、开源、现代化UI、支持动态追加、独立账号记忆**。

## 3. 技术原理深度分析

### 3.1 微信单实例限制机制

微信PC客户端通过以下四层机制确保单实例运行：

```
┌─────────────────────────────────────────────────┐
│              微信单实例防御体系                    │
├─────────────────────────────────────────────────┤
│  第1层：进程级互斥 (Mutex)                       │
│  ├─ 启动时调用 CreateMutexW 创建命名互斥体        │
│  ├─ 互斥体名称: "WeChat_Global_Mutex"            │
│  │  或 "_WeChat_App_Instance_Identity_Mutex_Name"│
│  └─ 若互斥体已存在 → 激活已有窗口 → 退出当前进程   │
├─────────────────────────────────────────────────┤
│  第2层：配置级绑定                               │
│  ├─ config.dat 含 AES加密的 device_id + wxid     │
│  └─ 同一设备ID重复登录 → 服务端拒绝              │
├─────────────────────────────────────────────────┤
│  第3层：数据库级共享                             │
│  ├─ MsgIndex.db (SQLite WAL模式) 共享访问        │
│  └─ 并发写入 → WAL journal 损坏风险              │
├─────────────────────────────────────────────────┤
│  第4层：运行时检测                               │
│  ├─ 检查 IsDebuggerPresent()                    │
│  ├─ 检查 GetModuleHandleA("WeChatUWP.dll")      │
│  └─ 非标准路径启动 → TerminateProcess            │
└─────────────────────────────────────────────────┘
```

### 3.2 多开核心技术方案对比

#### 方案A：关闭已创建的Mutex句柄（推荐 ⭐）

**原理**：枚举微信进程的所有句柄，找到目标Mutex对象，调用`CloseHandle`关闭它，使后续启动的微信实例认为没有已运行实例。

**流程**：
```
1. 定位 WeChat.exe 进程 PID
2. 调用 NtQuerySystemInformation 枚举进程句柄
3. 筛选类型为 Mutex 的句柄
4. 调用 NtQueryObject 获取句柄名称
5. 匹配目标 Mutex 名称（如 "WeChat_Global_Mutex"）
6. 调用 DuplicateHandle 复制句柄到当前进程
7. 调用 CloseHandle 关闭复制的句柄
8. 启动新的 WeChat.exe 实例
```

**优势**：
- 不修改微信原始二进制文件
- 不需要注入DLL
- 兼容性较好，微信更新后Mutex名称通常不变
- 可在微信已运行后动态追加新实例

**劣势**：
- 需要管理员权限
- 句柄枚举操作较底层

#### 方案B：API Hook拦截CreateMutexW

**原理**：通过DLL注入或Detours库，Hook微信进程的`CreateMutexW`函数，修改其行为使其始终返回新句柄（而非已存在互斥体的句柄）。

**优势**：
- 精确控制，可针对特定函数
- 不修改磁盘文件

**劣势**：
- DLL注入可能被杀毒软件拦截
- 微信更新后Hook点可能偏移
- 实现复杂度较高

#### 方案C：竞态条件（Race Condition）

**原理**：利用微信"检查Mutex"到"创建Mutex"之间的时间窗口，并发启动多个实例，使它们都认为没有已运行实例。

**优势**：
- 实现最简单
- 不需要管理员权限

**劣势**：
- 成功率不稳定，依赖系统负载
- 无法在已运行微信后动态追加
- 不推荐用于生产环境

#### 方案D：沙盒隔离

**原理**：利用Windows Sandbox或Sandboxie为每个微信实例创建完全隔离的运行环境。

**优势**：
- 完全物理隔离，零冲突
- 安全性最高

**劣势**：
- 资源消耗大（每个沙盒需1-2GB内存）
- 需要Windows专业版
- 数据持久化复杂

### 3.3 方案选择结论

**械式微信多开器采用方案A（关闭Mutex句柄）作为核心方案**，原因如下：

1. ✅ 支持"动态追加"——已打开微信后仍可开新窗口
2. ✅ 不修改微信原始文件，合规性较好
3. ✅ 不需要DLL注入，杀毒软件兼容性好
4. ✅ 兼容性稳定，Mutex名称跨版本变化少
5. ✅ 实现复杂度适中

辅助方案：结合**独立数据目录重定向**，确保每个微信实例的数据完全隔离。

## 6. 核心模块详细设计

### 6.1 Mutex管理器 (MutexManager)

**职责**：定位并关闭微信进程的Mutex句柄，使新实例得以启动。

**核心逻辑**：

```rust
pub struct MutexManager;

impl MutexManager {
    pub fn close_wechat_mutex(pid: u32) -> Result<(), String> {
        // 1. 打开目标进程
        // 2. 枚举进程所有句柄 (NtQuerySystemInformation)
        // 3. 筛选 Mutex 类型句柄
        // 4. 获取句柄名称 (NtQueryObject)
        // 5. 匹配微信 Mutex 名称
        // 6. 复制句柄到当前进程 (DuplicateHandle)
        // 7. 关闭句柄 (CloseHandle)
    }
}
```

**微信Mutex名称列表**（需持续更新）：

| 微信版本 | Mutex名称 |
|---------|----------|
| 3.x | `_WeChat_App_Instance_Identity_Mutex_Name` |
| 4.0+ | `WeChat_Global_Mutex` |
| 企业微信 | `Tencent.WeWork.ExclusiveObject` |

### 6.2 进程管理器 (ProcessManager)

**职责**：管理微信进程的启动、监控和终止。

**核心功能**：

| 功能 | 说明 |
|------|------|
| `launch_wechat()` | 启动新的微信实例 |
| `get_wechat_processes()` | 获取所有运行中的微信进程 |
| `monitor_process()` | 监控微信进程状态变化 |
| `terminate_instance()` | 终止指定微信实例 |
| `get_wechat_path()` | 从注册表获取微信安装路径 |

**数据目录重定向策略**：

每个微信实例使用独立的数据目录，避免配置冲突。核心原理：通过设置子进程环境变量 `USERPROFILE`，将微信实例的数据读写重定向到独立目录。

**实现方案：环境变量重定向**

微信PC客户端将用户数据存储在 `%USERPROFILE%\Documents\WeChat Files\` 下。通过 `CreateProcess` 或 `std::process::Command` 设置自定义 `USERPROFILE` 环境变量，可让每个微信实例将数据写入独立路径。

**重定向后的目录结构**：

```
%APPDATA%\械式微信多开器\
├── instances\
│   ├── ins_{uuid1}\
│   │   ├── Documents\              ← 模拟 %USERPROFILE%\Documents
│   │   │   └── WeChat Files\       ← 微信数据目录（独立）
│   │   ├── AppData\                ← 模拟 %USERPROFILE%\AppData
│   │   └── config.dat              ← 微信配置
│   ├── ins_{uuid2}\
│   │   ├── Documents\
│   │   │   └── WeChat Files\
│   │   └── ...
│   └── ...
└── app.db                         ← 账号数据库
```

**环境变量重定向核心实现**：

```rust
use std::path::PathBuf;
use std::process::Command;
use std::fs;

pub fn prepare_instance_dir(instance_id: &str) -> Result<PathBuf, String> {
    let base_dir = get_app_data_dir()?;
    let instance_dir = base_dir.join("instances").join(instance_id);
    
    // 创建必要子目录结构
    fs::create_dir_all(instance_dir.join("Documents").join("WeChat Files"))
        .map_err(|e| format!("创建实例目录失败: {}", e))?;
    fs::create_dir_all(instance_dir.join("AppData").join("Roaming"))
        .map_err(|e| format!("创建AppData目录失败: {}", e))?;
    
    Ok(instance_dir)
}

pub fn launch_wechat_with_redirect(
    wechat_path: &str,
    instance_dir: &PathBuf,
) -> Result<u32, String> {
    // 获取当前系统环境变量
    let current_userprofile = std::env::var("USERPROFILE")
        .unwrap_or_else(|_| String::new());
    
    let child = Command::new(wechat_path)
        // 设置 USERPROFILE 指向实例独立目录
        .env("USERPROFILE", instance_dir)
        // 传递原来的 TEMP 和 TMP 变量（避免临时文件路径问题）
        .env("TEMP", format!("{}\\Temp", instance_dir.display()))
        .env("TMP", format!("{}\\Temp", instance_dir.display()))
        .spawn()
        .map_err(|e| format!("启动微信失败: {}", e))?;
    
    Ok(child.id())
}
```

**关键注意事项**：

1. **首次启动兼容**：首次启动时实例目录为空，微信的配置文件（`config.dat`）不存在，微信会自动创建新配置，进入扫码登录界面。
2. **后续启动记忆**：再次以同一 `instance_dir` 启动微信时，微信读取已存储的 `config.dat`，扫码登录窗口会显示上次登录的账号头像，用户扫码后自动登录。
3. **数据完全隔离**：不同实例之间的聊天记录、文件缓存、配置均存储在不同目录，互不干扰。
4. **临时目录也需要隔离**：`TEMP` / `TMP` 变量也需要指向实例独立目录，否则微信的临时文件（如更新包、缓存）可能冲突。

### 6.3 实例标签管理 (InstanceLabelManager)

**职责**：管理微信实例的标签备注，帮助用户区分不同实例的用途。**不需要单独的账号管理功能**——微信自身的扫码登录机制已经通过独立数据目录实现了"记住账号"。

**设计原理**：

**数据模型**：


## 11. 核心代码实现参考

### 11.1 Mutex关闭核心实现 (Rust)

> **⚠️ 安全提示**：`DUPLICATE_CLOSE_SOURCE` 会关闭微信进程中的源句柄。微信后续若试图操作该句柄会产生未定义行为。本项目严格控制使用范围，仅关闭名称匹配的 Mutex 句柄，不影响其他句柄。

#### 11.1.1 NtQuerySystemInformation FFI 绑定

`NtQuerySystemInformation` 是未文档化的 NT API，需手动 FFI 绑定：

```rust
use windows::Win32::System::Threading::*;
use windows::Win32::Foundation::*;
use windows::Win32::Security::*;
use windows::Win32::System::SystemInformation::*;

#[repr(C)]
struct SystemHandleTableEntryInfo {
    unique_process_id: u16,    // 实际上为 ULONG_PTR，但低16位足够标识PID
    creator_back_trace_index: u16,
    object_type_index: u8,     // 对象类型索引（非直接类型编号）
    handle_attributes: u8,
    handle_value: u16,         // 实际上为 ULONG_PTR
    object: *mut std::ffi::c_void,
    granted_access: u32,
}

// 更精确的结构体（对齐到实际内存布局）
#[repr(C)]
struct SystemHandleInformationEx {
    number_of_handles: u32,
    reserved: u32,
    handles: [SystemHandleTableEntryInfo; 1], // 变长数组
}

// 未文档化 NT API 外部声明
extern "system" {
    fn NtQuerySystemInformation(
        system_information_class: u32,
        system_information: *mut std::ffi::c_void,
        system_information_length: u32,
        return_length: *mut u32,
    ) -> i32; // NTSTATUS
}

const SYSTEM_HANDLE_INFORMATION: u32 = 16;
const STATUS_INFO_LENGTH_MISMATCH: i32 = 0xC0000004u32 as i32;
const STATUS_SUCCESS: i32 = 0;
```

#### 11.1.2 完整句柄枚举与Mutex关闭

采用**先枚举所有Mutex类型句柄、后批量关闭**的策略，避免 `NtQueryObject` 查询名称可能导致的死锁风险。由于微信3.x和4.x使用的Mutex名称不同，我们通过**尝试关闭所有微信进程的Mutex类型句柄**来确保兼容。

```rust
const WECHAT_MUTEX_NAMES: &[&str] = &[
    "_WeChat_App_Instance_Identity_Mutex_Name",  // 微信 3.x
    "WeChat_Global_Mutex",                        // 微信 4.0+
];

/// 获取所有运行中的微信进程PID列表（按PID降序排列，优先处理最新实例）
pub fn get_all_wechat_pids() -> Vec<u32> {
    use sysinfo::System;
    let mut sys = System::new();
    sys.refresh_processes();
    
    let mut pids: Vec<u32> = sys.processes()
        .iter()
        .filter(|(_, p)| {
            let name = p.name().to_lowercase();
            name == "wechat.exe" || name == "weixin.exe"
        })
        .map(|(pid, _)| pid.as_u32())
        .collect();
    
    // 按PID降序排列（新启动的实例PID更大）
    pids.sort_by(|a, b| b.cmp(a));
    pids
}

/// 关闭所有微信进程的Mutex句柄（确保多次多开场景正确工作）
pub fn close_all_wechat_mutexes() -> Result<u32, String> {
    let pids = get_all_wechat_pids();
    if pids.is_empty() {
        return Ok(0); // 没有运行中的微信，无需关闭Mutex
    }
    
    let mut total_closed = 0u32;
    
    // 遍历所有微信进程，关闭每个进程的Mutex句柄
    for &pid in &pids {
        match close_mutexes_for_process(pid) {
            Ok(count) => total_closed += count,
            Err(e) => log::warn!("关闭进程 {} 的Mutex失败: {}", pid, e),
        }
    }
    
    Ok(total_closed)
}

/// 枚举指定进程的所有句柄并关闭Mutex类型句柄
fn close_mutexes_for_process(target_pid: u32) -> Result<u32, String> {
    unsafe {
        let process = OpenProcess(
            PROCESS_QUERY_INFORMATION | PROCESS_DUP_HANDLE,
            false,
            target_pid,
        ).map_err(|e| format!("无法打开进程 {}: {:?}", target_pid, e))?;

        // 第一步：查询系统句柄信息
        let mut buffer_size: u32 = 0x100000; // 初始1MB
        let mut closed_count = 0u32;
        const MAX_RETRIES: u32 = 3;
        
        for retry in 0..MAX_RETRIES {
            let mut buffer: Vec<u8> = vec![0u8; buffer_size as usize];
            let mut return_length: u32 = 0;
            
            let status = NtQuerySystemInformation(
                SYSTEM_HANDLE_INFORMATION,
                buffer.as_mut_ptr() as *mut std::ffi::c_void,
                buffer_size,
                &mut return_length,
            );
            
            if status == STATUS_INFO_LENGTH_MISMATCH {
                // 缓冲区不够大，扩大后重试
                buffer_size = return_length * 2;
                continue;
            }
            
            if status != STATUS_SUCCESS {
                CloseHandle(process).ok();
                return Err(format!("NtQuerySystemInformation 失败: 0x{:X}", status as u32));
            }
            
            // 解析句柄表头
            let header = &*(buffer.as_ptr() as *const SystemHandleInformationEx);
            let handle_count = header.number_of_handles as usize;
            
            // 第二步：遍历所有句柄，筛选属于目标进程的Mutex类型句柄
            let handles_ptr = buffer.as_ptr()
                .add(std::mem::size_of::<u32>() * 2)  // 跳过 header 的两个 u32
                as *const SystemHandleTableEntryInfo;
            let handles = std::slice::from_raw_parts(handles_ptr, handle_count);
            
            // Mutex 类型索引通常在 24-27 之间（取决于系统版本）
            // 对象类型编号可通过 PROCESS_TYPE_SNAPSHOT 动态确定
            // 此处采用范围筛选 + 名称验证的混合策略
            let mut closed_this_round = 0u32;
            
            for entry in handles.iter() {
                let entry_pid = entry.unique_process_id as u32;
                if entry_pid != target_pid {
                    continue;
                }
                
                // 对象类型索引 24-27 通常为 Mutant（互斥体）
                // 此处使用较宽泛的范围避免遗漏
                if entry.object_type_index < 20 || entry.object_type_index > 30 {
                    continue;
                }
                
                // 第三步：尝试通过 DuplicateHandle + NtQueryObject 验证是否为微信Mutex
                // 使用两步策略避免死锁：
                //   a) 先复制句柄到当前进程（不带 CLOSE_SOURCE）
                //   b) 在当前进程上下文中查询句柄名称
                
                let mut dup_handle = HANDLE::default();
                if DuplicateHandle(
                    process,
                    HANDLE(entry.handle_value as isize),
                    GetCurrentProcess(),
                    &mut dup_handle,
                    0,
                    false,
                    DUPLICATE_SAME_ACCESS,  // 注意：不带 CLOSE_SOURCE
                ).is_err() {
                    continue;
                }
                
                // 第四步：在当前进程上下文中安全查询句柄名称
                let is_wechat_mutex = check_if_wechat_mutex_name(dup_handle);
                
                if is_wechat_mutex {
                    // 确认是微信Mutex后，关闭目标进程中的源句柄
                    let mut close_source_handle = HANDLE::default();
                    if DuplicateHandle(
                        process,
                        HANDLE(entry.handle_value as isize),
                        GetCurrentProcess(),
                        &mut close_source_handle,
                        0,
                        false,
                        DUPLICATE_CLOSE_SOURCE | DUPLICATE_SAME_ACCESS,
                    ).is_ok() {
                        CloseHandle(close_source_handle).ok();
                        closed_this_round += 1;
                    }
                }
                
                CloseHandle(dup_handle).ok();
            }
            
            closed_count += closed_this_round;
            break; // 成功处理，退出重试循环
        }
        
        CloseHandle(process).ok();
        Ok(closed_count)
    }
}

/// 安全查询句柄名称：先在当前进程调用 NtQueryObject
/// 避免在目标进程上下文查询导致的死锁风险
fn check_if_wechat_mutex_name(handle: HANDLE) -> bool {
    // 方案A：尝试用 OpenMutexW 测试已知名称（更安全）
    for name in WECHAT_MUTEX_NAMES.iter() {
        unsafe {
            let name_wide: Vec<u16> = name.encode_utf16().chain(std::iter::once(0)).collect();
            let result = OpenMutexW(
                MUTEX_ALL_ACCESS,
                false,
                windows::core::PCWSTR(name_wide.as_ptr()),
            );
            if let Ok(mutex_handle) = result {
                CloseHandle(mutex_handle).ok();
                return true; // 该名称的Mutex存在
            }
        }
    }
    
    // 方案B：通过 NtQueryObject 查询句柄名称（作为备用）
    // 注意：此 API 可能在某些场景下阻塞，添加超时保护
    if let Ok(name) = get_handle_name_via_nt_query_object(handle) {
        for mutex_name in WECHAT_MUTEX_NAMES.iter() {
            if name.contains(mutex_name) {
                return true;
            }
        }
    }
    
    false
}
```

#### 11.1.3 DUPLICATE_CLOSE_SOURCE 安全性说明

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| **仅复制不关闭源** | 只用 `DUPLICATE_SAME_ACCESS`，不关微信内的源句柄 | ✅ 安全，不破坏微信状态<br>❌ Mutex 仍有引用计数，Mutex对象不会销毁 |
| **复制后关闭源** | 使用 `DUPLICATE_CLOSE_SOURCE` 关闭源句柄 | ✅ 可彻底销毁Mutex对象<br>⚠️ 微信若再次操作该句柄会出错 |
| **本项目采用** | 仅对已验证为微信Mutex的句柄使用 `DUPLICATE_CLOSE_SOURCE` | ✅ 精准打击，最小化风险 |

降低风险的额外措施：
1. **仅关闭Mutex类型句柄**，不触碰文件句柄、窗口句柄等
2. **关闭后立即启动新实例**，新实例创建新Mutex后旧进程的受影响窗口极小
3. 后续可引入**降级策略**：若 `DUPLICATE_CLOSE_SOURCE` 方案不稳定，可切换为仅关闭副本（纯 `DUPLICATE_SAME_ACCESS`），配合竞态启动策略

```rust
    pub fn launch_new_instance(&self, label: Option<&str>) -> Result<LaunchInfo, String> {
        let wechat_path = self.get_wechat_path()?;

        // 1. 生成实例ID
        let instance_id = uuid::Uuid::new_v4().to_string();

        let existing_hwnds = get_wechat_window_hwnds();

        let child = Command::new(&wechat_path)
            .env("USERPROFILE", &instance_dir)
            .env("TEMP", instance_dir.join("Temp"))
            .env("TMP", instance_dir.join("Temp"))
            .spawn()
            .map_err(|e| format!("启动微信失败: {}", e))?;

        let pid = child.id();

        // 6. 等待新窗口出现（差集定位）
        let hwnd = wait_for_new_wechat_window(&existing_hwnds, 30)?;

        self.db.insert_instance_full(
            &instance_id,
            label.unwrap_or(&format!("实例 {}", &instance_id[..8])),
            pid,
            &hwnd,
            &instance_dir.to_string_lossy(),

            pid,
            hwnd,
        })
```

```tsx
import { invoke } from '@tauri-apps/api/core';
import { Button, Card, message, Tag, List, Input } from 'antd';
import { useEffect, useState } from 'react';

interface LaunchInfo {
  instance_id: string;
  pid: number;
  hwnd: string;
}

interface InstanceInfo {
  id: string;
  label: string;
  pid: number;
  hwnd: string;
  data_path: string;
  status: 'running' | 'stopped';
  created_at: string;
}

export function ControlPanel() {
  const [instances, setInstances] = useState<InstanceInfo[]>([]);
  const [label, setLabel] = useState('');

  const handleLaunch = async () => {
    try {
      const result = await invoke<LaunchInfo>('launch_wechat', {
        label: label || null,
      });
      message.success(`微信启动成功，实例: ${result.instance_id}`);
      await refreshInstances();
    } catch (err) {
      message.error(`启动失败: ${err}`);
    }
  };

  const refreshInstances = async () => {
    const list = await invoke<InstanceInfo[]>('get_wechat_instances');
    setInstances(list);
  };

  useEffect(() => { refreshInstances(); }, []);

  return (
    <Card title="微信多开控制">
      <div style={{ marginBottom: 16 }}>
        <Input
          placeholder="实例标签（如：工作号）"
          value={label}
          onChange={e => setLabel(e.target.value)}
          style={{ width: 200, marginRight: 8 }}
        />
        <Button type="primary" size="large" onClick={handleLaunch}>
          打开新微信
        </Button>
      </div>
      <List
        dataSource={instances.filter(i => i.status === 'running')}
        renderItem={item => (
          <List.Item>
            <Tag color="green">运行中</Tag>
            {item.label} — PID: {item.pid}
          </List.Item>
        )}
      />
    </Card>
  );
}
```

### 16.1 微信Mutex名称发现方法

如需在新版微信中确认Mutex名称，可使用以下方法：

1. **Process Explorer**（微软官方工具）：
   - 运行微信 → 打开Process Explorer → 找到WeChat.exe进程
   - 双击进程 → 查看"Handles"标签 → 筛选"Mutant"类型
   - 记录Mutex名称

2. **WinObj**（微软Sysinternals工具）：
   - 运行微信 → 打开WinObj → 查看\BaseNamedObjects目录
   - 搜索"WeChat"相关条目

3. **代码枚举**：
   - 使用NtQuerySystemInformation枚举进程句柄
   - 使用NtQueryObject获取句柄名称
   - 筛选类型为Mutant的句柄
